# Membership Anomaly Detection — Business Flow & ML Architecture

> Scope: the Membership Anomaly Detection feature (*Deteksi Anomali Kepesertaan*, route
> `/analitik/anomali`, within the *Analitik & AI* section) in **open-jkn** — what it does
> end-to-end, how the machine-learning pipeline actually works (grounded in the code), and
> how it does (and does **not**) relate to **openIMIS**.

---

## TL;DR

- `/analitik` is a **JKN/BPJS enrollment anomaly-detection** feature.
- It is powered by a **bespoke Python ML pipeline** that lives in [`ai/`](../../ai/) — a
  **Keras autoencoder + scikit-learn Isolation Forest + hand-written business rules**,
  combined into a hybrid scorer.
- **It does NOT use any openIMIS AI/ML module.** openIMIS has no such module in its
  standard distribution. The analytics run on JKN data (the BPJS `PSTV*` schema); the
  only connection to openIMIS is the separate, unconditional `/sync` writer.
- The Node.js app talks to Python by **spawning a process per request** and exchanging
  CSV files through the OS temp dir. Results are **ephemeral** (not persisted to the DB).

---

## 1. Is this "openIMIS AI/ML"? (Direct answer)

**No.** This is the most important clarification.

| Question | Answer |
|---|---|
| Does openIMIS ship an AI/ML anomaly module? | **No.** openIMIS modules cover claims, contributions, enrolment, payments, feedback, etc. There is no standard "AI" or "ML" module. |
| What actually powers `/analitik`? | A custom Python package at [`ai/anomaly_detection/`](../../ai/anomaly_detection/) — built inside open-jkn, not imported from openIMIS. |
| What data does it model? | **JKN/BPJS kepesertaan (enrollment) data** using the BPJS sample variable schema `PSTV01`–`PSTV18` — *not* the openIMIS schema (`tblInsuree` / `tblFamilies` / `tblPolicy`). |
| Where does openIMIS appear in this feature? | Nowhere in the analytics path. openIMIS is only touched by the unrelated [`/sync`](../../app/api/sync/route.ts) feature, which writes participants into `tblInsuree`/`tblFamilies`/`tblPolicy`. |

So the honest framing is: **open-jkn analyzes JKN data with its own ML, and (separately) syncs JKN data into openIMIS.** Today those two concerns are **decoupled** — anomaly results are not fed into the sync decision (see §10).

## 2. Page structure & user journey

Three pages under [app/(app)/analitik/](../../app/(app)/analitik/):

```
/analitik               → landing/overview (explainer cards + rule reference)
/analitik/anomali       → the workspace: train / score (DB or file) / review results
/analitik/dashboard     → visualizations generated by the Python pipeline (PNGs)
```

**Landing page** ([page.tsx](../../app/(app)/analitik/page.tsx)) is purely informational — it
describes the three detection approaches (Autoencoder, Isolation Forest, Business Rules)
and lists the validation rules. It links into the other two.

**Anomaly workspace** ([anomali/page.tsx](../../app/(app)/analitik/anomali/page.tsx)) is where
work happens. It exposes three actions, each mapping to an API endpoint:

| UI action | Handler | Endpoint | Purpose |
|---|---|---|---|
| **Train Model** | `handleTrainModel` | `POST /api/ai/train` | Build model artifacts from DB participants |
| **Analyze Database** | `handleAnalyzeDatabase` | `POST /api/ai/analyze-database` | Score current DB participants using saved models |
| **Analyze File** | `handleAnalyzeFile` | `POST /api/ai/anomaly` | Score an uploaded `.csv` / `.dta` (BPJS sample) file |

On load it also calls `GET /api/ai/status` to check whether model artifacts exist and
whether output images are available (this drives the disabled/enabled state of the
Score buttons — you can't score until you've trained).

**Dashboard** ([dashboard/page.tsx](../../app/(app)/analitik/dashboard/page.tsx)) simply
renders the PNG charts emitted by Python (`anomaly_dashboard.png`,
`age_distribution.png`, `ae_training_history.png`) via
`GET /api/ai/images/[...slug]`, with graceful fallbacks when images don't exist yet.

Two distinct phases:

1. **Train** (`run_train.py`) — reads data, engineers features, fits the autoencoder +
   isolation forest, computes a threshold, and **saves artifacts** to
   `ai/anomaly_detection/saved_models/`. Also scores the training set for evaluation and
   emits charts.
2. **Score** (`run_score.py`) — loads saved artifacts, runs the same feature pipeline on
   **new** data, produces a hybrid anomaly score + rule flags, and writes
   `scored_new_data.csv` / `anomalies_new_data.csv` + charts. The Node layer reads those
   CSVs back and returns a summary + sliced rows to the UI.

## 4. Feature engineering ([features.py](../../ai/anomaly_detection/features.py))

Input columns are BPJS-coded (`PSTV01`…`PSTV18`) and renamed to readable names via
`COLUMN_MAP` in [config.py](../../ai/anomaly_detection/config.py) (e.g. `PSTV03`→`tanggal_lahir`,
`PSTV04`→`peran`, `PSTV08`→`jenis_peserta`).

`engineer_features()` derives:

- **`umur`** — age in years from `tanggal_lahir` vs a fixed `REFERENCE_DATE = 2023-01-01`.
- **`is_active`** — `status_peserta == "AKTIF"`.
- **Family-level features** (grouped by `id_keluarga`): `jml_keluarga`,
  `jml_keluarga_aktif`, `rasio_aktif`, `kepala_exists`, `jml_peran_unik`.
- **Cross-consistency flags**: `anak_tapi_dewasa` (role ANAK & age > 25),
  `kepala_tapi_muda` (head role & age < 17), `kawin_tapi_muda` (married & age < 16).
- **`kapitasi`** — coerced to numeric.

`encode_and_select()` then one-hot encodes the categoricals
(`peran`, `jenis_kelamin`, `status_kawin`, `kelas_rawat`, `jenis_peserta`) with
`drop_first=True`, concatenates the 11 numerical features, and returns the matrix.
During scoring it re-aligns columns to the training `feature_names` (missing → 0), which
is what lets a model trained on one dataset score a different one without crashing.

## 5. The three detectors

### 5.1 Autoencoder ([train.py](../../ai/anomaly_detection/train.py))

A symmetric dense autoencoder that learns to **reconstruct "normal" records**; high
reconstruction error ⇒ anomalous.

- Architecture: `input → Dense(64,relu)+BN+Dropout(0.2) → Dense(32) → Dense(16) →
  bottleneck Dense(8) → Dense(16) → Dense(32)+BN+Dropout → Dense(64)+BN →
  Dense(input_dim, sigmoid)`.
- Loss `mse`, optimizer `Adam(1e-3)`, **50 epochs**, batch 256, `EarlyStopping`
  (patience 10, restore best weights) + `ReduceLROnPlateau`.
- Score = mean squared reconstruction error per record.
- **Explainability**: per-feature squared errors are kept (`ae_feature_errors`) so each
  anomaly can show its **top-3 contributing features** (`top_ae_features`).

### 5.2 Isolation Forest ([train.py](../../ai/anomaly_detection/train.py))

- `n_estimators=200`, `contamination=0.01`, `n_jobs=-1`.
- Score = `-decision_function(...)` (sign-flipped so **higher = more anomalous**).

### 5.3 Business rules ([rules.py](../../ai/anomaly_detection/rules.py))

Domain heuristics split into **hard** (force `rule_flag = 1`) and **soft** (explanation
text only). See the table in §8.

## 6. Hybrid scoring & the final decision

Both raw scores are heavy-tailed and on different scales, so they are
**rank-normalized** to `[0,1]` and averaged:

```python
ae_ranks = rankdata(ae_scores) / n
if_ranks = rankdata(if_scores) / n
hybrid_scores = (ae_ranks + if_ranks) / 2
threshold = np.percentile(hybrid_scores, 99)   # top 1%
```

Final flag and provenance (see [train.py](../../ai/anomaly_detection/train.py) /
[score.py](../../ai/anomaly_detection/score.py)):

```
final_anomaly = (hybrid_score > threshold) OR (rule_flag == 1)

anomaly_source =
  "model+rule"  if model anomaly AND rule fired
  "model_only"  if model anomaly, no rule
  "rule_only"   if rule fired, no model anomaly
  "normal"      otherwise
```

This `anomaly_source` taxonomy is what the Node layer rolls up into the UI summary
(`ruleBased` / `modelBased` counts) in [analyze-database/route.ts](../../app/api/ai/analyze-database/route.ts)
and [anomaly/route.ts](../../app/api/ai/anomaly/route.ts).

## 7. Node ↔ Python bridge ([lib/ai-utils.ts](../../lib/ai-utils.ts))

The ML itself is Python; Node only orchestrates and presents.

- **`queryParticipantsForML()`** — queries `participant` (+ `familyMembers`) from the
  Postgres DB via Drizzle, maps each row to the BPJS `PSTV*` shape, and writes a CSV into
  `os.tmpdir()/openjkn-ai/uploads/`.
- **`runPythonScript()`** — `spawn(PYTHON_BIN, ["-u", script, ...args])` with
  `OPENJKN_AI_OUTPUT_DIR` injected via env. Captures stdout/stderr; rejects on non-zero
  exit. `PYTHON_BIN` defaults to `python3` and is overridable for deployments
  (the `PYTHON_BIN` commit made this configurable).
- **`parseCsvFile()`** — reads Python's output CSVs back into JS objects.
- The route handlers then compute a summary (`totalRecords`, `anomaliesCount`,
  `anomalyRate`, `ruleBased`, `modelBased`) and return **sliced** rows (100 scored / 50
  anomalies) plus image URLs.

> Note: the Python interpreter is **re-spawned for every request** and reloads the Keras
> model each time — there is no in-process model cache in Node. This is simple but means
> a noticeable cold-start cost per analysis (see §11).

## 8. Business rules reference

From [rules.py](../../ai/anomaly_detection/rules.py):

| Code | Type | Trigger |
|---|---|---|
| `KELUARGA_>50_ANGGOTA` | **Hard** | family size > 50 |
| `AKTIF_UMUR_>110` | **Hard** | active participant older than 110 |
| `UMUR_NEGATIF` | **Hard** | negative age (future birth date) |
| `KEPALA_KELUARGA_ANAK` | **Hard** | head role (`PESERTA`/`SUAMI`/`ISTRI`) under age 12 |
| `TANPA_KEPALA_KELUARGA` | Soft | family with > 1 member but no head |
| `RASIO_AKTIF_RENDAH` | Soft | active ratio < 0.2 in a family > 5 |
| `KELUARGA_BESAR` | Soft | family size > 10 |
| `ANAK_TAPI_PBPU` | Soft | child role but `PBPU` segment |
| `KAWIN_UMUR_<16` | Soft | married under 16 |
| `ANAK_TAPI_UMUR_>25` | Soft | child role over 25 |
| `AKTIF_LANSIA_>90` | Soft | active participant aged 90–110 |
| `KAPITASI_SANGAT_TINGGI` | Soft | kapitasi above the 99th percentile |

## 9. Artifacts & outputs

**Saved models** (`ai/anomaly_detection/saved_models/`, checked by
[`/api/ai/status`](../../app/api/ai/status/route.ts)):

- `autoencoder.keras` — the Keras model
- `isolation_forest.joblib` — scikit-learn model
- `scaler.joblib` — `MinMaxScaler`
- `feature_names.joblib` — training column order (for scoring alignment)
- `threshold.joblib` — the 99th-percentile cutoff

**Outputs** (`outputs_enrollment/`, also written to `os.tmpdir()` at runtime):

- `scored_new_data.csv`, `anomalies_new_data.csv` — consumed by Node
- `anomaly_dashboard.png`, `age_distribution.png`, `ae_training_history.png` — served to
  the dashboard via `/api/ai/images/[...slug]`

Stack: `tensorflow`, `scikit-learn`, `pandas`, `numpy`, `scipy`, `joblib`, `matplotlib`
(see [requirements.txt](../../ai/requirements.txt)).

## 10. Relationship to openIMIS (and the current gap)

Conceptually the pieces *could* form a pipeline:

```
JKN/BPJS data  →  (1) ML anomaly screening  →  (2) clean/validated data  →  (3) /sync → openIMIS
```

i.e. analytics could act as a **data-quality gate** before records are pushed into
openIMIS. **But the code does not wire this together today:**

- `/sync` ([route.ts](../../app/api/sync/route.ts)) syncs a participant **unconditionally** —
  it never consults anomaly scores or `final_anomaly`. A flagged record is synced exactly
  like a clean one.
- Anomaly results are **not persisted** anywhere (no DB table, no link to
  `participant`). They live only in temp CSVs for the duration of a request and are
  re-derived on every run.
- The two features even operate on **different schemas**: analytics uses the BPJS
  `PSTV*` shape; sync maps to the openIMIS `tbl*` shape.

So today the relationship is: **two independent features sharing the JKN domain**, not an
integrated "validate-then-sync" flow. Building that integration would be a product
decision (e.g. persist `anomaly_source`/`hybrid_score` on `participant`, and gate or
annotate `/sync` accordingly).

## 11. Engineering observations & gaps

Grounded-in-code notes worth knowing if you evolve this feature:

1. **DB export is lossy vs. the real BPJS sample.**
   [`transformToMLFormat`](../../lib/ai-utils.ts) emits only 10 of the 18 `PSTV*` columns
   (drops province, faskes, employer type, etc.). A model trained on a full `.dta` sample
   will have one-hot columns that don't exist in a DB export; scoring still works
   (missing → 0) but those features contribute nothing.

2. **`kapitasi` is effectively dead for DB-sourced data.** The DB exporter writes
   `PSTV15` as `"YA"/"TIDAK"` (an active *flag*), but `features.py` reads `kapitasi` as a
   **numeric amount** → `"YA"` parses to `NaN` → `fillna(0)` → always 0. So
   `KAPITASI_SANGAT_TINGGI` and the `kapitasi` feature only do anything on real BPJS file
   uploads, not on the "Analyze Database" path. (This is also a semantic mismatch:
   `PSTV15` is repurposed as a flag while the column name implies a currency value.)

3. **`peran` of the family head is hardcoded.** In the DB exporter the head's role is
   always `PISA_CODE_TO_ROLE["1"] = "PESERTA"` regardless of actual relationship, and
   family members default to `status_kawin = "BELUM_KAWIN"`. Consequence: rules that
   depend on precise member marital status (e.g. `KAWIN_UMUR_<16`) won't fire for
   members on the DB path.

4. **Fixed reference date.** `REFERENCE_DATE = 2023-01-01` ages are computed against a
   frozen date, so `umur` drifts further from reality over time. Age-threshold rules
   (`>110`, `<16`, `<12`) inherit this drift.

5. **Ephemeral, stateless results.** Nothing is stored — re-running overwrites the temp
   CSVs/PNGs. There's no audit trail, no per-participant anomaly history, and the
   dashboard images reflect only the most recent run.

6. **Per-request Python spawn + TF load.** Each `/api/ai/*` call cold-starts a Python
   process and reloads Keras. Fine for an internal/admin tool, painful at any real
   concurrency. A long-lived Python sidecar (or loading artifacts once) would help.

7. **Upload validation is extension-based.** [anomaly/route.ts](../../app/api/ai/anomaly/route.ts)
   checks the file extension (`csv`/`dta`) and a small allowlist of MIME types, but not
   true content validation. The path is passed as a process **argument** (not a shell),
   so there's no command-injection risk — but malformed CSVs surface as generic 500s.

8. **Explainability is a real strength.** The `anomaly_source` taxonomy plus
   `top_ae_features` (top-3 reconstruction-error drivers) give reviewers concrete reasons
   per flag — better than a bare anomaly score.

## 12. How to run

```bash
# 1. Python deps (once)
pip install -r ai/requirements.txt

# 2. Train from the DB (or drop a BPJS .dta/.csv in and use "Analyze File")
#    Either click "Train Model" in /analitik/anomali, or:
python ai/run_train.py <optional/path/to/data.csv>

# 3. Score — via the UI ("Analyze Database" / "Analyze File"), or directly:
python ai/run_score.py <path/to/data.csv|dta>

# In deployments where python3 isn't on PATH, tell the Node side where it is:
#   PYTHON_BIN=/usr/bin/python3.11   (in .env / env)
```

Outputs land in `ai/outputs_enrollment/` (or the OS temp dir at runtime); saved models
land in `ai/anomaly_detection/saved_models/`.
